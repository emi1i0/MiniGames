# City Bloxx (Skyline)

Toy tower builder in **Three.js** (cartoon low-poly, near-profile 2.5D camera). A crane swings the next apartment floor back and forth over the top of the tower as a pendulum; tap / click / space releases it and it flies along its swing arc onto the stack — the block **keeps the horizontal velocity it had at release**, so it drops fast near the center of the swing and nearly straight down at the extremes. The game is about **timing and balance**: floors are full width, so a drop lands wherever its arc carried it, and every off-center floor shifts the tower's center of mass. Drift the COM past the base footprint and the whole tower topples — run over. A drop whose offset hangs the block past its outermost-window line also ends the run (see the landing rule below). Score is **habitantes** (population), not points: a clean placement fills a floor; a crooked one houses fewer people. Each building has a floor cap; reaching it **completes the building** for a population bonus and the next floor simply begins a new building in a new color (no roof piece — that just looked odd). This replaced the original plain-canvas version.

See `DESIGN.md` ("Ciudad de Juguete") for the art direction; every visual answers to it.

## Module layout

The four systems the rework is organized around:

- `game/Crane.ts` — **Module 1, pendulum physics.** The block hangs from a fixed jib pivot at x = 0 above the tower top and swings as `angle = amp * sin(t*omega)` plus an erratic wind gust (two incommensurate sines). `omega`, `amp` and the wind all grow with the floor count, so the swing gets faster and less predictable as you climb. Owns the jib / cable / hook meshes; `blockX` / `blockY` is the hanging block's world position, and `blockVx` is its live horizontal velocity (finite-differenced from the arc — fast near the center, ~0 at the extremes). On drop the pendulum is disabled and `Game` free-falls the block while keeping that inherited `blockVx` (times `DROP_INHERIT`), so the trajectory is a real sideways-plus-down parabola.
- `game/Tower.ts` — **Module 2, collision / offset / balance.** The stacked floors, box-overlap collision against the top floor, the alignment `offset`, the center-of-mass, and the **whip sway**. Each floor keeps a structural `restX`; its *visible* X adds a sway offset that grows with height (the whip), driven by a static lean toward the COM, a transient wobble spring kicked on each off-center drop, and a continuous wind sway that appears once the COM drifts. `place()` resolves a drop (perfect snaps dead-center and rigid; crooked stays where it landed and kicks the wobble; `|offset| > MAX_LANDING_OFFSET` returns `{ ok: false }` = a fall). `isToppled()` + `detachFloors()` end the run.
- `game/Environment.ts` + the camera in `Game.ts` — **Module 3, dynamic camera + backdrop.** The camera lerps its vertical focus to follow the tower top (`updateCamera`), and the sun/shadow frustum tracks it. `Environment` lerps the sky color and fog through street level -> clouds -> stratosphere -> black space keyed to the tower-top height, fades a starfield in up high, and drifts low clouds past. Pure eye-candy; touches no gameplay.
- `game/Particles.ts` + the combo / population logic in `Game.ts` — **Module 4, combos & population.** Gold additive spark bursts on perfect / combo / building-complete (`Particles`). `Game` tracks `perfectCombo`; `COMBO_THRESHOLD` (3) consecutive perfects light the combo state, which doubles population (`COMBO_MULT`) and speeds up the swing (`COMBO_SPEEDUP`).

Supporting files:

- `main.ts` — entry point, mounts `Game` into `#app`.
- `game/Game.ts` — scene / camera / renderer, an `EffectComposer` with a subtle bloom pass, lights + soft shadows, the `ready / countdown / playing / dead` state machine, the drop flow, population, building bookkeeping, missed-block debris, and the render loop (`tick` -> `composer.render()`).
- `game/Blocks.ts` — the toy-block mesh factory: `buildFloor` (rounded enamel body + lit-window grid + ink outline), `buildFoundation`, and the per-building `BUILDING_COLORS` palette.
- `game/materials.ts` — the shared cartoon look copied from the repo's other 3D games: cached `toon(color)` (MeshToonMaterial + 4-step gradient), `emissiveToon` (glowing lit windows), `outlined(geo, mat)` (inverted-hull dark outline), `flatGeo`.
- `game/Hud.ts` — DOM overlay: population (the score is habitantes), remaining lives (dots), the balance meter, the current building progress, the combo glow, the tone-colored flash, and the start / game-over / countdown screens + leaderboard.
- `game/InputController.ts` — keyboard + pointer -> a single `onDrop`. **Pointer is bound on `window`** (not the canvas) so tap-to-start works while the full-screen overlay is on top, but taps on interactive chrome (`input, button, a, .leaderboard`) are ignored so the leaderboard form and back link stay usable.
- `game/SoundEffects.ts` — synthesized Web Audio effects (countdown tick, release whoosh, landing thud, perfect chime, miss honk (`playLifeLost`), building-complete fanfare (`playRoof`), collapse rumble). No assets.
- `game/constants.ts` — all tunable values (pendulum, drop gravity, balance/whip, population, building caps, lives, camera, height zones). **Tune here first.**

## Non-obvious decisions

**World units.** 1 unit == one floor's height; the foundation top is y = 0; the pendulum swings along X with Z fixed (the camera is near-profile 2.5D, so depth is cosmetic). Floor `i`'s center sits at `(i + 0.5) * FLOOR_H`.

**Balance, not stacking precision, is the fail state.** Unlike `stack-tower`, floors are never trimmed — they keep full width and stay where they land. `Tower.comOffset()` is the mean of the floors' structural `restX` minus the base; once `|comOffset| > TOPPLE_LIMIT` the COM is off the base and `isToppled()` ends the run. `MAX_LEAN` maps the balance ratio to a visible lean so the tower leans toward the danger side before it goes. (A single badly-placed first floor whose offset already exceeds the limit topples on its own — that is intended.)

**Topple = crumble, not a rigid rotation.** On topple `Game.crumble()` calls `Tower.detachFloors()` (which pulls every floor mesh out of the tower group and returns its world X/Y) and turns each into independent `debris` flung toward the topple side (`CRUMBLE_KICK`, growing with the floor's height for a whip) and popped up (`CRUMBLE_HOP`) with a tumble. This replaced the old rigid "rotate the whole tower group around the base" collapse, which **swept the floors straight through the base slab** (a floor at `centerY = 0.5` rotating about the base origin ends up stabbed into the foundation — the exact "muy ficticio" bug the player reported). The upward pop keeps even a near-center bottom floor clear of the slab (same reason as the miss bounce); verified with a debris-trajectory sim (zero slab intersection for every topple position). The building now falls apart off to the side and the base is left clean.

**The whip (why the top sways more than the base).** `Tower.worldXOf` displaces each floor by `sin(totalAngle) * centerY * (1 + WHIP_GAIN * centerY/height)` — a rotation around the base plus a height-weighted term, so upper floors overshoot lower ones like a whip. `totalAngle` is `lean + wobble + ambientSway`: the static COM lean, a damped angular spring kicked per off-center drop (`WOBBLE_*`), and a continuous wind sway. This is the brutal-at-altitude difficulty: the swinging block **and** the swaying top both move in X, so the collision uses `topWorldX()` sampled at the instant the block lands, and you must predict the crossing.

**The continuous sway scales with instability, not just height.** `ambientSway`'s amplitude is `min(1, |comOffset|/TOPPLE_LIMIT) * AMBIENT_SWAY_MAX * instability`, where `instability` is passed into `Tower.update(dt, instability)` by `Game`: `min(SWAY_INSTABILITY_MAX, max(0, tower.count - BUILDING_CAP_BASE) * SWAY_RAMP_PER_FLOOR)`. So it is **exactly 0 for the whole first building** (the base stays rock-solid however crooked) and then grows floor-by-floor from the 2nd building on. Combined with the `|comOffset|` factor, a **badly-built** tall tower sways more and more the higher it climbs, while a precisely-centered one stays calm — precision keeps the tower stable, sloppiness makes it worse at altitude.

**Perfect placement.** A drop within `PERFECT_OFFSET` of the floor below snaps its `restX` to the floor below's (`topRestX`) — a rigid, drift-free stack that can climb forever — and adds no wobble kick. A crooked drop keeps its landing X (its `restX` is the drop X minus the sway at that height, so it sits coherently in the swaying stack) and kicks the wobble spring proportionally to the offset.

**Landing rule: stack within the last-window line (`MAX_LANDING_OFFSET`).** A drop only lands if `|offset| <= MAX_LANDING_OFFSET`; otherwise it falls and the run ends. The threshold is **derived from the window layout**, not a magic number: `MAX_LANDING_OFFSET = FLOOR_W/2 - WINDOW_OUTER_EDGE`, where `WINDOW_OUTER_EDGE` (0.77) is the outer edge of the outermost window (computed from `WINDOW_COLS` / `WINDOW_W`, the same constants `Blocks.ts` uses to draw the windows). So the rule is exactly "the new block must overhang the floor below by no more than the margin outside its last window" — a bare corner touch no longer counts (that was the old permissive `MIN_OVERLAP = 0.18`, which allowed ~2.2 of offset). With the current geometry that's ~0.43 of a floor's 1.2 half-width. It is demanding but fair: a swing-timing sim shows dead-center is achievable and ~13.5% of the swing is a valid release window. **If you change the window layout, the landing tolerance moves with it** — keep `WINDOW_COLS`/`WINDOW_W` as the single source of truth.

**The base is a floor-width footprint.** `buildFoundation`'s load-bearing slab is exactly `FLOOR_W` wide (X) — the first floor is supported by the same width as every floor above, so the landing rule reads identically at the base and a missed/toppled block falls off the slab's side instead of clipping through a base wider than the tower. (The slab is padded in depth/Z only, which is not the play axis, for a foundation look.) This is what makes the miss-bounce and topple-crumble land cleanly beside the base for **every** offset, including near-center first-floor misses that the wider old base clipped.

**Population, not points (`Game.populationFor`).** A floor houses `BASE_POP`; a perfect adds `PERFECT_BONUS`; a crooked one scales down by placement quality (unstable = fewer residents). A lit combo doubles it. Completing a building adds a `BUILDING_BONUS_PER_FLOOR * cap` bonus on top of that floor's population. The leaderboard score is total habitants (`direction: "higher"`, the default — so `meta.ts` declares no `scoring`).

**Misses end the run.** A drop past the landing tolerance (`|offset| > MAX_LANDING_OFFSET`, above) is a miss: `onMiss` turns the block into free-falling `debris` and ends the run immediately (`die(false)`, tower left standing). There are **no lives** — precision is the whole game, so one whiff is fatal, same as toppling (which ends it with the crumble instead). The two fail states: miss the stack, or drift the COM off the base. The missed block is **bounced off the edge** (`onMiss`): it's popped up (`vy = -MISS_HOP`) and flung outward toward the side it missed on (`vx = dir * max(MISS_KICK, |blockVx|)`) with a tumble, so it rises clear of the base slab's top and arcs down beside it. The upward pop matters because at the miss instant the block's bottom is on the slab top, so any downward-biased deflection would intersect the slab for a few frames; only popping it *up* clears the slab's Y-extent before it descends. Together with the floor-width base (above), a debris-trajectory sim shows **zero** slab intersection for every miss position, including the near-center first-floor misses the strict rule now produces.

**Buildings (no roofs).** Each building allows `computeCap()` floors (`BUILDING_CAP_BASE` growing `BUILDING_CAP_GROWTH` per building, capped). Placing the floor that reaches the cap **completes** the building: `resolveDrop` pays a `BUILDING_BONUS_PER_FLOOR * cap` population bonus + fanfare + spark burst, then increments `buildingIndex` so the next `spawnBlock` uses the next `buildingColor`. There is deliberately **no roof mesh** — the building just switches color on completion (the old pitched-roof lid read as weird). Endless: a stack of colored buildings, each cap a milestone and a difficulty bump.

**Camera + shadows follow the top.** `updateCamera` lerps `camY` toward `tower.landingTopY()` and repositions the camera (near-profile: small `CAM_X`, large `CAM_Z`, narrow `CAM_FOV` ~ ortho) and the sun's shadow frustum so shadows stay crisp over the build area regardless of height.

**The block is a persistent mesh, so `resetWorld` must drop it.** Unlike the old canvas version (where the in-play block was plain data), the hook block is a real scene mesh. `resetWorld` removes `this.block`'s mesh before spawning the new one — otherwise the block spawned for the `ready` screen is orphaned in the scene when Enter restarts, leaving a second floor floating on the hook. Same reason `resetWorld` `clearDebris()`es and `tower.reset()` removes the floor meshes.

**Cable direction.** In `Crane.syncMeshes` the cable pivot rotates by `+angle` (not `-angle`): the block hangs at `+sin(angle)*CABLE_LEN`, so a negative rotation would swing the cable opposite the block and it would look detached. The camera focus sits `CAM_FOCUS_UP` **above** the tower top so the jib and hanging block stay framed.

**`dt` is clamped** (`MAX_DT`) so a tab-switch can't integrate one giant step and teleport a falling block through the stack.

**Enter-to-start countdown.** From the start / game-over screen, Enter / tap / space enters a `countdown` state showing 3 / 2 / 1 / YA (`COUNTDOWN_LABELS`, `COUNTDOWN_STEP` s each) with the shared 750 Hz tick before play begins. Mandatory shared pattern (see root `CLAUDE.md`).

## Room mode (multiplayer)

Wired to the shared party mode: the constructor calls `initRoomMode("city-bloxx", { getScore: () => this.population, onStart: () => this.beginCountdown() })` (see root `CLAUDE.md`, "Salas"). With `?room=` the game-over reports the population to the room instead of the global ranking, the restart input is blocked (one run per round), and the round auto-starts via `onStart`. `getScore` returns the live population for the timeout partial. Because a perfectly idle player would swing the crane forever without ever dropping, the game declares `roomTimeLimitSec: 180` in `meta.ts`. Without the param nothing changes.
